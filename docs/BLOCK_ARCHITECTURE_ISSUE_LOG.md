# Block Architecture Issue Log

## 목적

`block API + stable/experimental block` 원칙은 현재 프로젝트의 코드지옥을 줄이기 위한 기준이다.

하지만 이 원칙 역시 실제 프로젝트에서 검증되기 전까지는 실험적 성격을 가진다.

따라서 문제를 숨기거나 즉흥적으로 원칙을 바꾸지 않고, 이 파일에 문제와 처리 결과를 누적한다.

동일한 유형의 문제가 반복되면, 그때 원칙을 수정할지 검토한다.

## 운영 원칙

```text
1. 문제 발생 시 즉시 이 파일에 기록한다.
2. 문제를 한 번 겪었다고 바로 헌법을 바꾸지 않는다.
3. 동일 유형의 문제가 반복될 때 원칙 수정 검토 대상으로 올린다.
4. 수정이 필요하면 별도 결정 기록 후 헌법 문서에 반영한다.
5. 성공한 stable block은 문제 원인으로 확정되기 전까지 직접 수정하지 않는다.
```

## 문제 유형 분류

```text
A. Stable Block 훼손 위험
B. Experimental Block 격리 실패
C. callBlock 호출 규약 불편/누락
D. manifest 파라미터/반환형식 불일치
E. UI가 block 내부를 직접 만지는 문제
F. block이 너무 커져 또 다른 거대 파일이 되는 문제
G. 문서와 실제 코드 불일치
H. 성능 저하
I. 디버깅 난이도 증가
J. 기타
```

## 원칙 수정 검토 기준

동일 유형의 문제가 아래 조건 중 하나를 만족하면 원칙 수정 검토 대상으로 올린다.

```text
1. 같은 유형의 문제가 3회 이상 반복된다.
2. stable block 보존 원칙 때문에 명백한 개발 병목이 발생한다.
3. callBlock 구조 때문에 기능 구현이 불필요하게 복잡해진다.
4. 실험 block 격리가 오히려 중복 코드 폭증을 만든다.
5. manifest 유지 비용이 실제 이득보다 커진다.
6. UI/엔진/파서 간 경계가 현실적으로 유지되지 않는다.
```

## 누적 이슈

### 2026-05-17 / ISSUE-0001 / Block API 원칙 자체의 실험성 인정

- 유형: J
- 발생 위치: 설계 단계
- 관련 block: 전체
- 증상: `block API + stable/experimental block` 원칙이 아직 실제 구현으로 충분히 검증되지 않았음.
- 원인: 현재 구조는 코드지옥을 줄이기 위한 설계안이지만, 실제 대량 기능 확장 환경에서의 운영 비용은 아직 검증 전임.
- 임시 처리: 원칙을 바로 절대화하지 않고, 문제 발생 시 이 파일에 누적 기록하기로 함.
- 최종 처리: 동일한 유형의 문제가 반복되면 원칙 수정 여부를 검토한다.
- 재발 여부: 관찰 필요
- 원칙 수정 필요성: 관찰
- 메모: 헌법과 별개로 문제/처리결과를 누적하는 안전장치 파일을 둔다.

## 향후 검토 방식

1. 일정 시점마다 이 파일의 누적 이슈를 검토한다.
2. 동일 유형 문제가 반복되는지 확인한다.
3. 반복 문제가 실제 생산성을 떨어뜨리는지 확인한다.
4. 필요하면 `STANDARD_BLOCK_API_ARCHITECTURE.md` 또는 관련 기준 문서를 수정한다.
5. 수정 결정은 이 파일의 `원칙 수정 결정 기록`에 남긴다.
